E-commerce Microservices CDK Setup Guide

Quick Start

This guide will help you set up the CDK infrastructure for our e-commerce microservices workshop.

Prerequisites

  1. AWS CLI configured with appropriate permissions
  2. Node.js (v16 or later)
  3. AWS CDK installed globally
  4. Docker for building container images

Installation Steps

1. Install AWS CDK

npm install -g aws-cdk

2. Verify Installation

cdk --version

3. Bootstrap CDK (First time only)

cdk bootstrap aws://ACCOUNT-NUMBER/REGION

Project Structure

ecommerce-microservices-cdk/
├── bin/
│   └── app.ts                    # CDK App entry point
├── lib/
│   ├── stacks/
│   │   ├── network-stack.ts      # VPC, subnets, security groups
│   │   ├── ecr-stack.ts         # Container repositories
│   │   ├── ecs-stack.ts         # ECS cluster
│   │   ├── api-gateway-stack.ts  # API Gateway HTTP API
│   │   └── monitoring-stack.ts   # CloudWatch logs and metrics
│   └── constructs/
│       └── microservice.ts       # Reusable microservice construct
├── services/
│   ├── product-service/
│   ├── inventory-service/
│   ├── order-service/
│   ├── payment-service/
│   ├── supplier-service/
│   └── consumer-service/
├── cdk.json
├── package.json
└── README.md

CDK Commands

Deploy all stacks

cdk deploy --all

Deploy specific stack

cdk deploy NetworkStack

List all stacks

cdk list

Show differences

cdk diff

Synthesize CloudFormation

cdk synth

Destroy all resources

cdk destroy --all

Environment Variables

Create a .env file in the project root:

# AWS Configuration
AWS_ACCOUNT=123456789012
AWS_REGION=us-east-1

# Application Configuration
APP_NAME=ecommerce-microservices
ENVIRONMENT=dev

# Container Configuration
ECR_REPOSITORY_PREFIX=ecommerce
CONTAINER_CPU=256
CONTAINER_MEMORY=512

# Network Configuration
VPC_CIDR=10.0.0.0/16

Key CDK Constructs Used

1. Network Stack

  • VPC with public/private subnets
  • Internet Gateway and NAT Gateway
  • Security Groups for services
  • VPC Endpoints for AWS services

2. Container Stack

  • ECR repositories for each service
  • ECS Fargate cluster
  • Service discovery namespace
  • Task definitions and services

3. API Gateway Stack

  • HTTP API Gateway
  • VPC Link for private integration
  • Routes for each microservice
  • CORS configuration

4. Monitoring Stack

  • CloudWatch Log Groups
  • Custom metrics and dashboards
  • Alarms for critical metrics

Service Discovery

Services are registered automatically with AWS Cloud Map:

const service = new FargateService(this, 'ProductService', {
  cluster: props.cluster,
  taskDefinition,
  cloudMapOptions: {
    name: 'product-service',
    cloudMapNamespace: props.namespace,
  },
});

Inter-Service Communication

Services communicate using DNS names:

// Product service URL
const productServiceUrl = 'http://product-service.ecommerce.local:3000';

// API call
const response = await fetch(`${productServiceUrl}/api/products`);

Monitoring and Logging

All services automatically log to CloudWatch:

logging: LogDrivers.awsLogs({
  streamPrefix: 'product-service',
  logGroup: new LogGroup(this, 'ProductLogGroup', {
    logGroupName: '/ecs/product-service',
    retention: RetentionDays.ONE_WEEK,
  }),
}),

Security Best Practices

  1. IAM Roles: Least privilege access for each service
  2. Security Groups: Restricted network access
  3. VPC: Private subnets for services
  4. Secrets Manager: Sensitive configuration data

Troubleshooting

Common Issues:

  1. Bootstrap Required

    cdk bootstrap aws://ACCOUNT/REGION

  2. ECR Repository Already Exists

    • Delete existing repositories or use different names

  3. VPC Limits

    • Check VPC quotas in AWS console

  4. IAM Permissions

    • Ensure CDK has necessary permissions

Cost Optimization

  • Use Fargate Spot for non-production workloads
  • Enable auto-scaling for services
  • Set appropriate log retention periods
  • Use reserved capacity for predictable workloads
